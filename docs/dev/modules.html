Modules

At present modules are fairly loose mechanism for grouping Controllers and the routes that they require to be used within a Proem application into a self contained directory stucture.

Eventually, this mechanism will be extended so that Modules will almost be considered stand alone applications that are able to be distributed and reused.

Modules are simply a collection of Controllers and a simple Module Loader class which is used to attach the Routes required for these controllers to be used. This Module Controller can also be used to load Plugins, Views and any other code the module might depend upon.

The easiest way to describe there functionality is probably to build a simple module. So, building on the application within the Quick Start Guide that is exactly what I intend to do here. In actual fact, within the Quick Start Guide we have already built a simple Index module. The Module we will build here however will also show us how we can start to move the code required to Bootstrap a Module out of our main bootstrap index.php file and into somewhere more self contained.

The directory structure

Firstly, we need a directory structure within which we will place our Module. Go to the root of the myproject application and create the following directories:

mkdir -p Module/Foo/Controller

The Module Loader

Within Module/Foo we now need to create our actual Module class: 

    namespace Module\Foo;

    class Foo extends \Proem\Ext\Module\Generic
    {
        public function init(\Proem\Service\Manager\Template $assets, $environment = null)
        {
            $assets->get('events')->attach('proem.post.in.router', function($event) {
                $event->getServiceManager()->get('router')->attach(
                    'foo-route', new \Proem\Routing\Route\Standard([
                        'rule'      => '/foo',
                        'targets'   => ['module' => 'foo', 'controller' => 'index', 'action' => 'index']
                    ])
                );
            });
        }
    }

If you have been reading the docs in the order present it is hopefully quite easy to see that all this code is doing is adding a Route to the router. For this simple Module we only need this single route, however, in real life Modules would be far more complex.

All Module Loaders must extend the generic \Proem\Ext\modile\Generic module loader abstract class and therefore must implement the init() method. The $enviroment variable is as yet not used, however it will be used in future version to allow switching from say dev to production handling.

It is important to note here that we are using the Service Manager that is passed in to our listener even though we already seem to have access to it via the $assets variable. This is because getting it from the listener like we are means it has already travelled it's way through most of the Bootstrap process. Using $assets here would be of little use as the actual assets we need would not exist yet.

For this reason, I often prefer to separate the callback into another method so as to resist temptation. I think it makes the code more readable too. 

      namespace Module\Foo;

      class Foo extends \Proem\Ext\Module\Generic
      {
          public function init(\Proem\Service\Manager\Template $assets, $environment = null)
          {
              $assets->get('events')->attach('proem.post.in.router', [$this, 'attachRoutes']);
          }

          public function attachRoutes($event) {
              $event->getServiceManager()->get('router')->attach(
                  'foo-route', new \Proem\Routing\Route\Standard([
                      'rule'      => '/foo',
                      'targets'   => ['module' => 'foo', 'controller' => 'index', 'action' => 'index']
                  ])
              );
          }
      }

Controllers

Just like the Index controller created within the Quick Start Guide our Module needs a controller. Place the following file within Module/Foo/Controller/Index.php: 

    namespace Module\Foo\Controller;

    class Index extends \Proem\Controller\Standard
    {
        public function indexAction()
        {
            $this->assets->get('response')->appendToBody('Hello from Foo!');
        }
    }

Loading Our Module

All that is left now is to load our Module. Edit the htdocs/index.php file so that it looks lie the following: 

    require_once '/Users/thorpe/www/lib/proem/framework/lib/Proem/Autoloader.php';

    (new Proem\Autoloader())
        ->attachNamespace('Module', '../lib')
        ->register();

    (new Proem\Proem)
        ->attachModule(new Module\Foo\Foo)
        ->attachEventListener('proem.post.in.router', function($event) {
            $event->getServiceManager()->get('router')->attach(
                'hello-route', new \Proem\Routing\Route\Standard([
                    'rule'      => '/hello/:name/:params',
                    'targets'   => ['module' => 'index', 'controller' => 'index', 'action' => 'hello']
                ])
            );
        }
    )->init();

Note the call to attachModule? This simple method loads our Module by calling it's init() method.

By default modules are loaded when the proem.init event is triggered. You can however load a Module latter in the bootstrap process by passing the name of the event as the second argument to attachModule. You can also priorities Modules using the third argument.

attachModule proxies through to a protected method on the Proem object which has a definition that looks like: 

    protected function attachExtension(Extension $extension, $event = 'proem.init', $priority = 0)

You can see from this definition that Modules are really Extensions. Plugins are the other type of extension which is the next topic within the docs.

Start your server now and head to http://127.0.0.1:8080/foo to see Hello from Foo!.

Summary

Hopefully this brief guide has at least described the potential of Modules. With these under your belt it's easy to see how we can remove a lot of setup from our bootstrap index.php file and into it's own Module which, with a little bit of work, should be pretty easy to then re use.
